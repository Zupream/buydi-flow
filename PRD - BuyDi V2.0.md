## 1. บทนำ (Introduction)

เอกสารกำหนดความต้องการผลิตภัณฑ์ (PRD) นี้กำหนดความต้องการเชิงหน้าที่และไม่ใช่เชิงหน้าที่สำหรับ BuyDi โดยทำหน้าที่เป็นแหล่งข้อมูลหลัก (Single Source of Truth) สำหรับสิ่งที่จะสร้างและให้ทีมติดตามได้จากความต้องการถึงการดำเนินการ

BuyDi เป็นแพลตฟอร์มตลาดกลางทั่วไป (General Marketplace) ที่เปิดโอกาสให้ผู้ประกอบการและผู้ขายไทย (SME) ลงทะเบียนและจำหน่ายสินค้าออนไลน์ ระบบถูกออกแบบมาสำหรับทั้งนิติบุคคลและผู้ขายรายบุคคล โดยมอบโครงสร้างพื้นฐานอีคอมเมิร์ซที่สมบูรณ์พร้อมการดำเนินงานที่โปร่งใสและเครื่องมือจัดการหลังบ้านระดับมืออาชีพ

เอกสารนี้ครอบคลุม Phase 1 ทั้งหมด รวมถึงการอัปเดตจาก Production ล่าสุด

## 2. สรุปผู้บริหาร (Executive Summary)

BuyDi เป็นแพลตฟอร์มตลาดกลางทั่วไป (รองรับทุกหมวดหมู่สินค้า) ที่เปิดโอกาสให้ผู้ประกอบการและผู้ขายไทย (SME) ลงทะเบียนและจำหน่ายสินค้าออนไลน์ แพลตฟอร์มถูกออกแบบมาสำหรับทั้งนิติบุคคล (ธุรกิจที่มีหนังสือรับรองบริษัท) และผู้ขายรายบุคคล (ใช้บัตรประชาชน) โดยมอบโครงสร้างพื้นฐานอีคอมเมิร์ซที่สมบูรณ์พร้อมการดำเนินงานที่โปร่งใสและเครื่องมือจัดการหลังบ้านระดับมืออาชีพ

**กลุ่มเป้าหมาย:** ผู้ขาย SME ไทย (ธุรกิจขนาดเล็กถึงขนาดกลาง) และผู้ขายรายบุคคลทุกหมวดหมู่สินค้า เพื่อให้บริการตลาดกลาง B2C ในประเทศไทย

**จุดแตกต่างหลัก:** BuyDi ช่วยให้ผู้ขายบรรลุการเติบโตทางอีคอมเมิร์ซอย่างยั่งยืน ด้วยการมอบเครื่องมือจัดการหลังบ้านที่ละเอียดและแม่นยำ (เช่น การจัดการสต็อกและตัวเลือกสินค้า) พร้อมด้วยความโปร่งใสทางการเงินที่สมบูรณ์ (การแสดงรายได้สุทธิและค่าคอมมิชชั่นที่ชัดเจน) ต่างจากคู่แข่งอย่าง Shopee และ Lazada ที่ BuyDi เน้นความโปร่งใสในการดำเนินงานด้วยการแสดงค่าธรรมเนียมที่ชัดเจน เครื่องมือหลังบ้านที่แข็งแกร่ง สำหรับการจัดการสินค้าคงคลัง กระบวนการอนุมัติที่คาดการณ์ได้ (24-48 ชั่วโมง) และการควบคุมแบรนด์ที่เข้มแข็งสำหรับผู้ขาย

## 3. วิสัยทัศน์และเป้าหมายของผลิตภัณฑ์ (Product Vision & Goals)

### 3.1 วิสัยทัศน์ (Product Vision)

สร้างตลาดกลางออนไลน์ที่โปร่งใสที่สุดสำหรับผู้ประกอบการไทย ด้วย Ecosystem ที่เชื่อมทุก Role เข้าด้วยกัน — Buyer ซื้อ → Seller ขาย → Affiliate โปรโมท → Admin ดูแล

### 3.2 วัตถุประสงค์ทางธุรกิจ

1. **การเติบโตของผู้ขาย** — ดึงดูดและรักษาผู้ขาย SME ที่จริงจังซึ่งต้องการเครื่องมือจัดการหลังบ้านระดับมืออาชีพ มุ่งเน้นคุณภาพมากกว่าปริมาณ
2. **GMV และความสามารถในการทำกำไร** — ขับเคลื่อนปริมาณการขายผ่านความโปร่งใสทางการเงินที่สร้างความไว้วางใจ สร้างรายได้ค่าคอมมิชชั่นที่ยั่งยืน
3. **ความพึงพอใจของผู้ใช้** — รักษาอัตราการรักษาผู้ขายที่สูงผ่านการดำเนินงานที่โปร่งใสและระบบที่เชื่อถือได้ บรรลุความพึงพอใจของผู้ซื้อผ่านสินค้าคุณภาพและการปฏิบัติตามที่เชื่อถือได้
4. **ส่วนแบ่งตลาด** — สร้างตำแหน่งที่แข็งแกร่งในกลุ่ม SME ไทย กลายเป็นที่รู้จักในฐานะตลาดกลางที่โปร่งใสที่สุดสำหรับผู้ประกอบการไทย
5. **ความเป็นเลิศในการดำเนินงาน** — ลดข้อผิดพลาดผ่านเครื่องมือหลังบ้านที่แม่นยำ (การจัดการสต็อก, ระบบ SKU) รักษา SLA การอนุมัติ 24-48 ชั่วโมงอย่างสม่ำเสมอ

### 3.3 ตัวชี้วัดความสำเร็จ (KPIs)

**การเงิน/การเติบโต:**

- GMV รายเดือน (มูลค่าสินค้าขั้นต้น) เป็นเงินบาทไทย
- รายได้ค่าคอมมิชชั่นต่อเดือน
- มูลค่าคำสั่งซื้อเฉลี่ย (AOV)

**อุปทาน/ผู้ขาย:**

- จำนวนผู้ขายที่ใช้งานอยู่ (รายเดือน/รายไตรมาส)
- อัตราการรักษาผู้ขาย (%)
- จำนวนสินค้าที่ลงรายการ

**อุปสงค์/ผู้ซื้อ:**

- จำนวนผู้ซื้อที่ใช้งานอยู่ (รายเดือน/รายไตรมาส)
- อัตราการรักษาผู้ซื้อ (%)

**ประสิทธิภาพ/คุณภาพ:**

- อัตราการแปลงผู้ขาย (ผู้ขายที่อนุมัติ / ผู้สมัครทั้งหมด)
- เวลาเฉลี่ยในการดำเนินการสถานะคำสั่งซื้อ (รอดำเนินการ → จัดส่งแล้ว)
- เวลาเฉลี่ยในการเผยแพร่สินค้า (เวลาอนุมัติ: เป้าหมาย 24-48 ชั่วโมง)
- เวลาตอบสนอง API < 200ms สำหรับ 95% ของคำขอ
- เวลาโหลดหน้า < 2 วินาที (first contentful paint)
- ระบบพร้อมใช้งาน 99.9%

## 4. กลุ่มผู้ใช้งานเป้าหมาย (Target Audience & User Roles)

### 4.1 Buyer (ผู้ซื้อ)

- ต้องมีบัญชีลูกค้าเพื่อทำการซื้อ
- ข้อมูลที่มองเห็นได้: ชื่อลูกค้า, ที่อยู่จัดส่ง, หมายเลขโทรศัพท์, สถานะบัญชี, ประวัติการสั่งซื้อ
- คาดหวังการติดตามการจัดส่งที่เชื่อถือได้ (หมายเลขติดตาม, วันที่จัดส่งโดยประมาณ)
- ชื่นชม product variations ที่ชัดเจนและข้อมูลสต็อกที่แม่นยำ
- **Journey:** สินค้า(Aff Link) → Add to Cart → Checkout → ชำระเงิน → ติดตามออเดอร์ → รับสินค้า → Order Completed

### 4.2 Seller / Vendor (ผู้ขาย)

- **ขนาดธุรกิจ:** Micro (1-5 คน) ถึง Small/Medium (สูงสุด 100 พนักงาน)
- **ประเภท:** นิติบุคคล (หนังสือรับรองบริษัท) และผู้ขายรายบุคคล (บัตรประชาชน)
- **ระดับทักษะทางเทคนิค:** ระดับกลางถึงขั้นสูง
- **Pain Points ปัจจุบัน:** ขาดความโปร่งใสทางการเงิน, โครงสร้างค่าธรรมเนียมซับซ้อน, เครื่องมือจัดการสต็อกไม่เพียงพอ
- **ความต้องการ:** มองเห็นกำไรสุทธิที่ชัดเจน, เครื่องมือจัดการสินค้าคงคลังที่แข็งแกร่ง, กระบวนการอนุมัติที่คาดการณ์ได้
- **Journey:** สมัคร → Admin Review → Approved → ลงสินค้า → Admin ตรวจ → เปิดขาย → รับ Order → Pack → ส่ง → Delivered → Hold 3d → Revenue Released → ถอนเงิน

### 4.3 Affiliate / Creator (ครีเอเตอร์)

- สมัครผ่านระบบ → ต้องผ่านการอนุมัติจาก Admin
- สร้าง Affiliate Link → แชร์/โปรโมท → User คลิก → Cookie Attribution (30 วัน) → User ซื้อ → Order Completed → Commission คำนวณ → Withdraw
- **Attribution Model:** Last Click — คลิกสุดท้ายก่อนซื้อได้ Commission ทั้งหมด

### 4.4 Admin (ผู้ดูแลระบบ)

- **บทบาท:** การดำเนินงานแพลตฟอร์มและการควบคุมคุณภาพ
- **ความรับผิดชอบ:**
  1. การลงทะเบียนผู้ขาย: ตรวจสอบและอนุมัติ/ปฏิเสธภายใน SLA 24-48 ชั่วโมง
  2. การจัดการสินค้า: ตรวจสอบคุณภาพ ข้อมูล ราคา ก่อนเผยแพร่
  3. การจัดการทางการเงิน: กำหนดอัตราค่าคอมมิชชั่น, ประมวลผลคำขอถอนเงิน
  4. ความปลอดภัยของเอกสาร: จัดเก็บเอกสารยืนยันตัวตนอย่างปลอดภัย
  5. การกำหนดค่าระบบ: ตั้งค่า parameters, workflows, เกณฑ์ต่างๆ
  6. การแก้ไขข้อพิพาท: ตัดสินข้อพิพาทระหว่าง Buyer-Seller (ผลถือเป็นที่สิ้นสุด)
  7. การจัดการตลาด: Flash sales, ระบบ affiliate, คูปอง

## 6. สถาปัตยกรรมและข้อกำหนดทางเทคนิค (Architecture & Technical Specs)

### 6.1 แนวคิดสถาปัตยกรรม

สถาปัตยกรรมแบบ Modular Monolith บน Laravel 12.x + Botble CMS พร้อม RESTful API v1 สำหรับการเชื่อมต่อระบบภายนอก

### 6.2 องค์ประกอบทางเทคโนโลยีหลัก

| Component | Technical Details |
|-----------|-------------------|
| **Framework** | Laravel 12.x + Botble CMS |
| **Database** | MySQL/MariaDB |
| **Cache** | Redis |
| **Authentication** | Laravel Sanctum (token-based) + Social Login (Facebook, Google, LinkedIn, GitHub) |
| **API** | RESTful API v1 (OpenAPI/Swagger) |
| **Cloud** | AWS/GCP/Azure |
| **Email** | SendGrid/Mailgun |
| **SMS/OTP** | Twilio/AWS SNS |

### 6.3 External Dependencies

**Payment Gateways (9 รายการ):**
Stripe (พร้อม Stripe Connect), PayPal, Razorpay, Mollie, SSL Commerz, Paystack, Pay Solutions, SCB, COD

**พันธมิตรการจัดส่ง:**

- GetShip (การจัดการบัญชีผู้ค้า, การสร้างการจัดส่ง, การติดตาม)
- Shippo (อัตราหลายผู้ให้บริการ, การตรวจสอบที่อยู่)
- ไปรษณีย์ไทย (Thai Post)

**Social Login:** Facebook OAuth, Google OAuth, LinkedIn OAuth, GitHub OAuth

### หมวดที่ 1: การจัดการผู้ขาย (4 FRs)

#### FR-001: การลงทะเบียนและการตรวจสอบผู้ขาย

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** เปิดให้ทั้งนิติบุคคลและผู้ขายรายบุคคลลงทะเบียนบนแพลตฟอร์ม พร้อมระบบตรวจสอบเอกสารและอนุมัติภายใน 24-48 ชั่วโมง

**ขอบเขตในรุ่นนี้:**

- รองรับการลงทะเบียนนิติบุคคลพร้อมอัปโหลดหนังสือรับรองบริษัท (อายุไม่เกิน 6 เดือน), ภพ.20 (ถ้ามี), หน้าสมุดบัญชีในนามบริษัท
- รองรับการลงทะเบียนบุคคลธรรมดาพร้อมอัปโหลดบัตรประชาชนและรูปถ่ายคู่กับบัตร (Selfie)
- ยอมรับรูปแบบเอกสาร: PDF, JPG, JPEG, PNG, WEBP (สูงสุด 2MB)
- ชื่อบัญชีธนาคาร **ต้องตรง** กับชื่อผู้สมัครหรือชื่อบริษัทเท่านั้น
- Admin สามารถตรวจสอบและอนุมัติ/ปฏิเสธใบสมัครภายใน 24-48 ชั่วโมงทำการ
- จัดเก็บเอกสารยืนยันตัวตนทั้งหมดแบบเข้ารหัสและปลอดภัย (เฉพาะ Admin เท่านั้น)
- ส่งการแจ้งเตือนไปยังผู้ขายเมื่ออนุมัติ/ปฏิเสธ
- ผู้ขายที่ได้รับการอนุมัติสามารถเข้าถึงแดชบอร์ดผู้ขาย

**เกณฑ์ยอมรับ:**

- ผู้ขายสามารถลงทะเบียนได้ทั้ง 2 ประเภท (นิติบุคคล/บุคคลธรรมดา)
- ระบบตรวจสอบชื่อบัญชีธนาคารตรงกับข้อมูลผู้สมัคร
- Admin ได้รับแจ้งเตือนเมื่อมีใบสมัครใหม่
- SLA 24-48 ชั่วโมงสามารถวัดผลได้

#### FR-002: การปรับแต่งร้านค้าและการสร้างแบรนด์

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** ให้ผู้ขายปรับแต่งรูปลักษณ์ร้านค้าด้วยโลโก้, แบนเนอร์, เนื้อหาร้านค้า และลิงก์โซเชียลมีเดีย พร้อม HTML editor สำหรับการสร้างเนื้อหาร้านค้าขั้นสูงและกำหนดค่า SEO

**ขอบเขตในรุ่นนี้:**

- อัปโหลดโลโก้แบบกำหนดเอง (แนะนำ 300x300 พิกเซล)
- อัปโหลดแบนเนอร์แบบกำหนดเอง (แนะนำ 1920x400 พิกเซล)
- HTML editor สำหรับการสร้างเนื้อหาร้านค้า
- กำหนดค่า SEO: SEO Title, Description, URL Slug
- URL slug ต้องใช้เฉพาะตัวอักษรภาษาอังกฤษ, ตัวเลข, เครื่องหมายขีดกลาง (ห้ามใช้ภาษาไทย)
- เพิ่มลิงก์โซเชียลมีเดีย (Facebook, Instagram, ฯลฯ)
- กำหนดข้อมูลสาขา/สำนักงานใหญ่

#### FR-003: แดชบอร์ดและการวิเคราะห์ของผู้ขาย

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** แดชบอร์ดที่ครอบคลุมแสดงแผนภูมิรายได้, ตัวชี้วัดประสิทธิภาพ, รายงานการขาย และการวิเคราะห์ธุรกิจ

**ขอบเขตในรุ่นนี้:**

- แสดงแผนภูมิรายได้แสดงแนวโน้มการขายตามช่วงเวลา
- แสดงแดชบอร์ดตัวชี้วัดประสิทธิภาพ (ยอดดู, การแปลง, ฯลฯ)
- สร้างรายงานการขายพร้อม Total Sales, Number of Orders, AOV
- ฟังก์ชันส่งออกรายงาน (PDF, CSV)
- กรองช่วงวันที่สำหรับการวิเคราะห์ย้อนหลัง

#### FR-004: การจัดการทางการเงินและการถอนเงิน

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** ความโปร่งใสทางการเงินที่สมบูรณ์พร้อมการแสดงรายได้รวม ค่าคอมมิชชั่น รายได้สุทธิ และระบบถอนเงิน

**ขอบเขตในรุ่นนี้:**

- แสดงแดชบอร์ดทางการเงิน: รายได้รวม, ค่าคอมมิชชัน, รายได้สุทธิ
- แสดงรายละเอียดค่าคอมมิชชันตามหมวดหมู่สินค้า
- **การคำนวณรายได้สุทธิ (อัปเดต v2.0):**
  - ฐานคำนวณ = ยอดหลังหักส่วนลดร้านค้า (ไม่รวมค่าส่ง / ส่วนลดแพลตฟอร์ม)
  - หัก: Platform Fee % + ค่าธุรกรรมการชำระเงิน % (ฐาน = ยอดที่ลูกค้าชำระ) + Affiliate Commission % + ส่วนต่างค่าส่ง (ถ้า Actual > Estimated)
  - = **รายได้สุทธิร้านค้า**
- **กรณี Cancel:** ค่าธุรกรรมการชำระเงิน Platform รับภาระแทน ไม่เรียกเก็บจาก Seller
- คำนวณค่าธรรมเนียมการถอนเงินล่วงหน้าก่อนส่ง
- แสดงจำนวนเงินที่ผู้ขายจะได้รับหลังหักแล้ว
- กำหนดค่าข้อมูลการจ่ายเงิน (ชื่อธนาคาร, ชื่อเจ้าของบัญชี, เลขบัญชี)
- ส่งคำขอถอนเงินพร้อมการตรวจสอบจำนวนขั้นต่ำ
- ติดตามสถานะการถอนเงิน (รอดำเนินการ, กำลังประมวลผล, เสร็จสมบูรณ์)
- รองรับวิธีการจ่ายเงินหลายแบบ (Stripe Connect, PayPal Payout, โอนเงินผ่านธนาคาร)

### หมวดที่ 2: การจัดการสินค้า (5 FRs)

#### FR-005: การสร้างและการจัดการสินค้า

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** ให้ผู้ขายสร้างและจัดการสินค้าพร้อมรายละเอียดครอบคลุม ต้องการการอนุมัติจาก Admin (24-48 ชั่วโมง) ก่อนเผยแพร่

**ขอบเขตในรุ่นนี้:**

- สร้างสินค้าพร้อมชื่อ, คำอธิบาย, ราคา
- อัปโหลดรูปภาพสินค้า (แนะนำ 800x800+ พิกเซล)
- เลือกหมวดหมู่สินค้าจากรายการที่กำหนดไว้ล่วงหน้า
- ตั้งค่า SEO Title, Description, URL Slug สำหรับหน้าสินค้า
- ส่งสินค้าเพื่อการอนุมัติจาก Admin
- Admin ตรวจสอบและอนุมัติ/ปฏิเสธภายใน 24-48 ชั่วโมงทำการ
- แก้ไขสินค้าที่อนุมัติแล้ว (ต้องอนุมัติใหม่หากมีการเปลี่ยนแปลงสำคัญ)
- ลบสินค้า (soft delete เพื่อรักษาประวัติคำสั่งซื้อ)
- ตั้งค่าราคาลดสำหรับการแสดงส่วนลด
- **Commission Setting:** ผู้ขายกำหนดค่าคอมมิชชันแยกตามรายสินค้าได้ (เช่น สินค้า A ให้ 5%, สินค้า B ให้ 10%)

#### FR-006: ตัวเลือกสินค้าและการจัดการ SKU

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** รองรับตัวเลือกสินค้า (เช่น ขนาด, สี) พร้อม SKU, ราคา และสต็อกแยกต่างหากสำหรับแต่ละตัวเลือก

**ขอบเขตในรุ่นนี้:**

- สร้างตัวเลือกสินค้า (เช่น ขนาด: S/M/L, สี: แดง/น้ำเงิน/เขียว)
- กำหนด SKU ที่ไม่ซ้ำกันต่อตัวเลือก (เช่น "TSHIRT-RED-M")
- ตัวเลือกสร้าง SKU อัตโนมัติพร้อมรูปแบบที่ปรับแต่งได้
- ตั้งราคาแยกต่างหากสำหรับแต่ละตัวเลือก
- ตั้งจำนวนสต็อกแยกต่างหากสำหรับแต่ละตัวเลือก
- ผู้ซื้อสามารถเลือกตัวเลือกเฉพาะเมื่อสั่งซื้อ

#### FR-007: การจัดการสินค้าคงคลังพร้อมการแจ้งเตือนสต็อกเหลือน้อย

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** การจัดการสต็อกที่ละเอียดในระดับตัวเลือกพร้อมการอัปเดตสต็อกอัตโนมัติ การแจ้งเตือนสต็อกเหลือน้อย และการป้องกันการขายเกินจำนวน

**ขอบเขตในรุ่นนี้:**

- ติดตามจำนวนสต็อกในระดับตัวเลือก (ไม่ใช่แค่ระดับสินค้า)
- ลดสต็อกอัตโนมัติเมื่อมีการสั่งซื้อ
- คืนสต็อกอัตโนมัติเมื่อมีการยกเลิกคำสั่งซื้อ
- กำหนดค่าเกณฑ์สต็อกเหลือน้อยต่อสินค้า/ตัวเลือก
- ส่งการแจ้งเตือนเมื่อสต็อกต่ำกว่าเกณฑ์
- ป้องกันการขายเกินโดยบล็อกคำสั่งซื้อเมื่อสต็อก = 0

#### FR-008: การจัดการสินค้าดิจิทัล

**ลำดับความสำคัญ:** Should Have

**วัตถุประสงค์:** รองรับการขายสินค้าดิจิทัลพร้อมการสร้างรหัสใบอนุญาตอัตโนมัติและการส่งมอบเมื่อสั่งซื้อ (ไม่ต้องจัดส่ง)

#### FR-009: ระบบเวอร์ชันสินค้า

**ลำดับความสำคัญ:** Should Have

**วัตถุประสงค์:** ติดตามการเปลี่ยนแปลงทั้งหมดที่ทำกับข้อมูลสินค้า พร้อมประวัติเวอร์ชันสำหรับการตรวจสอบและการย้อนกลับ

### หมวดที่ 3: การจัดการคำสั่งซื้อ (3 FRs)

#### FR-010: การประมวลผลและการจัดการสถานะคำสั่งซื้อ

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการวงจรชีวิตคำสั่งซื้อทั้งหมดตั้งแต่การสั่งซื้อจนถึงการจัดส่ง

**ขอบเขตในรุ่นนี้:**

- รับคำสั่งซื้อพร้อมข้อมูลผู้ซื้อ (ชื่อ, ที่อยู่, โทรศัพท์, สินค้าที่สั่ง)
- **Order Status State Machine (อัปเดต v2.0):**
  - **Happy Path:** Pending Payment → Pending → Processing → Shipped → Delivered → Completed
  - **Branching States:**
    - Cancelled: Buyer/Seller ยกเลิกก่อน Processing หรือ Timeout ไม่ชำระภายในเวลา
    - Return Flow: Return Approval → Waiting Product → Product Returned → Return Success
    - Debate (ไกล่เกลี่ย): เมื่อ Seller ไม่อนุมัติคืนสินค้า → Admin ตัดสิน
- อัปเดตสถานะคำสั่งซื้อด้วยตนเองโดยผู้ขาย
- ลดสต็อกอัตโนมัติเมื่อยืนยันคำสั่งซื้อ / คืนสต็อกอัตโนมัติเมื่อยกเลิก
- ส่งการแจ้งเตือนให้ผู้ซื้อเมื่อเปลี่ยนสถานะ
- **Escrow (อัปเดต v2.0):** ระบบถือเงินไว้ **3 วัน** หลัง Delivered → Auto-complete เมื่อพ้นกำหนด

#### FR-011: การเชื่อมต่อการจัดส่งและการติดตาม

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** เชื่อมต่อกับพันธมิตรการจัดส่ง (GetShip, Shippo) สำหรับการสร้างการจัดส่ง, การสร้างเลขพัสดุ และการอัปเดตสถานะการจัดส่งแบบเรียลไทม์

**ขอบเขตในรุ่นนี้:**

- กำหนดค่าบัญชี GetShip merchant ในการตั้งค่าผู้ขาย
- สร้างการจัดส่งผ่านการเชื่อมต่อ GetShip
- สร้างเลขพัสดุอัตโนมัติหรือป้อนด้วยตนเอง
- อัปเดตคำสั่งซื้อพร้อมเลขพัสดุและวันที่ส่งมอบโดยประมาณ
- ติดตามการอัปเดตสถานะการจัดส่งแบบเรียลไทม์
- แสดงสถานะการจัดส่งให้ผู้ซื้อ (ระหว่างขนส่ง, กำลังนำส่ง, ส่งมอบแล้ว)
- **SLA การจัดส่ง:** ผู้ขายควรนำส่งสินค้าให้บริษัทขนส่งภายใน 24-48 ชม.

#### FR-012: การจัดการการคืนสินค้าและข้อพิพาท

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการคำขอคืนสินค้าและข้อพิพาทพร้อมเวิร์กโฟลว์หลายขั้นตอน

**ขอบเขตในรุ่นนี้ (อัปเดต v2.0 — ปรับปรุงสำคัญ):**

**เงื่อนไข:** Order = Delivered + ภายใน **3 วัน** + Buyer แนบหลักฐาน

**ขั้นตอนที่ 1 — Buyer ยื่นคำร้อง:**

- ผู้ซื้อสามารถร้องขอคืนสินค้า/คืนเงินพร้อมเหตุผล
- ส่งหลักฐาน (รูปภาพ, คำอธิบาย)

**ขั้นตอนที่ 2 — Seller ตรวจคำขอ (3 วัน, ไม่ตอบ = Auto เลือก A):**

| ทางเลือก | รายละเอียด |
|----------|-----------|
| **A: คืนสินค้า + คืนเงิน** | Buyer ส่งคืนภายใน 3 วัน → Seller ตรวจของ (3 วัน ไม่ตรวจ = auto approve) → Seller รับผิดชอบค่าส่ง 2 ขา → Admin คืนเงิน Buyer |
| **B: คืนเงินอย่างเดียว** | ไม่ต้องคืนสินค้า → Seller สละสิทธิ์รับสินค้าคืน → Admin คืนเงิน Buyer ทันที |
| **C: ไม่อนุมัติ** | Seller ใส่เหตุผล → Buyer โต้แย้งได้ใน 3 วัน → ไม่โต้แย้ง = ปิดเคส (Order Complete) → โต้แย้ง = Dispute |

**ขั้นตอนที่ 3 — Dispute (Admin ตัดสิน — ผลถือเป็นที่สิ้นสุด):**

| ผลตัดสิน | สินค้าอยู่ที่ Buyer | สินค้าอยู่ที่ Seller |
|----------|---------------------|---------------------|
| **Buyer Win** | คืนเงินเต็ม + Buyer เก็บสินค้า | คืนเงินเต็ม + Seller รับค่าส่ง 2 ขา |
| **Seller Win** | ไม่คืนเงิน + Buyer เก็บสินค้า | ไม่คืนเงิน + Seller รับค่าส่งขากลับ 1 ขา |

**ช่องทางการคืนเงิน:**

| ช่องทางที่ชำระมา | วิธีการคืนเงิน | ระยะเวลา |
|-----------------|---------------|---------|
| บัตรเครดิต/เดบิต | คืนวงเงินเข้าบัตรเดิม (Void/Refund) | 7-15 วันทำการ |
| QR PromptPay / COD | โอนเงินคืนเข้าบัญชีธนาคาร | 3-5 วันทำการ หลังอนุมัติ |

### หมวดที่ 4: การตลาดและโปรโมชั่น (3 FRs)

#### FR-013: การจัดการ Flash Sales

**ลำดับความสำคัญ:** Should Have

**วัตถุประสงค์:** ผู้ขายกำหนดค่า flash sales พร้อมราคาตามเวลา จำนวนจำกัด Admin จัดการ flash sales ทั่วแพลตฟอร์มข้ามผู้ขายหลายราย

#### FR-014: การจัดการคูปองและส่วนลด

**ลำดับความสำคัญ:** Should Have

**วัตถุประสงค์:** ผู้ขายสร้างคูปองเฉพาะผู้ขายพร้อมรหัสส่วนลด, ส่วนลดเปอร์เซ็นต์/จำนวนคงที่, ช่วงเวลา, ขีดจำกัดการใช้งาน Platform Coupons สร้างโดย Admin (แพลตฟอร์มเป็นผู้สนับสนุนค่าใช้จ่าย)

#### FR-015: ระบบการตลาดแบบพันธมิตร (Affiliate System)

**ลำดับความสำคัญ:** Could Have

**ขอบเขตในรุ่นนี้ (อัปเดต v2.0):**

### หมวดที่ 9: Operational Flows — ใหม่ใน v2.0 (6 FRs)

#### FR-031: ระบบจัดส่ง Non-COD และตรวจน้ำหนัก (Shipping Logic & Weight Check)

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการระบบจัดส่งแบบชำระเงินล่วงหน้าพร้อมตรวจสอบน้ำหนักจริง vs ประมาณ และออกเอกสารที่เกี่ยวข้องอัตโนมัติ

**ขอบเขตในรุ่นนี้:**

- **ขั้นตอน:** Seller ยืนยันจัดส่ง → ขนส่งรับพัสดุ → ตรวจน้ำหนัก (GetShip Webhook)
- **Decision Tree น้ำหนัก:**
  - น้ำหนักตรง (Actual = Estimated): ออก RS Document → ส่งปกติ
  - Actual > Estimated: Seller จ่ายส่วนต่าง → ออก RSA Document → หักจาก Seller Wallet → แจ้ง Seller
  - Actual < Estimated: ส่วนต่าง = BuyDi Revenue → ไม่ต้องแจ้ง Seller
- **ส่งสำเร็จ:** Delivered → Completed (3 วัน) → Credit to Seller Wallet
- **ส่งไม่สำเร็จ / ตีกลับ:** คืนสินค้าให้ Seller → ยกเลิก/คืนเงิน Buyer → ออก RST + RS VOID

**เกณฑ์ยอมรับ:**

- ระบบรับ Webhook จาก GetShip เพื่ออัปเดตน้ำหนักจริง
- เอกสาร RS/RSA ออกอัตโนมัติตามเงื่อนไข
- ส่วนต่างค่าส่งหักจาก Seller Wallet โดยอัตโนมัติ

#### FR-032: ระบบเก็บเงินปลายทาง COD (COD Shipping & Reconciliation)

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการระบบ COD ตั้งแต่การจัดส่งจนถึงการกระทบยอดเงินที่ขนส่งโอนกลับ

**ขอบเขตในรุ่นนี้:**

- **Flow:** COD Order → จัดส่ง → Buyer จ่ายเงิน COD → ขนส่งเก็บเงิน → ปณ.โอน COD ให้ BuyDi → Generate Transaction ID + Validate → Update Status → Waiting COD Settlement → Order Completed → หักค่าขนส่ง → Available for Withdrawal
- **การจัดการความเสี่ยง COD:**
  - ปฏิเสธรับพัสดุ: ส่งคืน Seller — ค่าส่งไป-กลับหัก Seller
  - Remittance Delay: ขนส่งโอนช้า 3-7 วัน ระบบต้อง track
  - ยอดไม่ตรง: Auto-flag + Manual review ถ้ายอดไม่ match
  - ยกเลิก COD: ไม่มี Refund (ยังไม่จ่ายเงิน)

**เกณฑ์ยอมรับ:**

- ระบบสร้าง Transaction ID อัตโนมัติเมื่อรับเงิน COD
- ยอดเงิน COD ถูก validate กับยอดคำสั่งซื้อ
- Auto-flag เมื่อยอดไม่ตรง

#### FR-033: Commission Lifecycle State Machine

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการวงจรชีวิตของ Commission ตั้งแต่เกิดจนถอนเงินได้

**ขอบเขตในรุ่นนี้:**

- **State Machine:** Create Order → Order Completed → อนุมัติ → ได้ Balance → ถอนเงิน
- **Exceptions:**
  - Order Cancelled → ไม่อนุมัติ Commission
  - ถอนเงิน: Admin อนุมัติ (จ่ายแล้ว) หรือ Admin ไม่อนุมัติ (คืน Balance)
- รอบการจ่ายเงิน: ตามรอบที่บริษัทกำหนด (เช่น ทุกวันที่ 15 และ 30 ของเดือน)

**เกณฑ์ยอมรับ:**

- Commission ถูกสร้างอัตโนมัติเมื่อ Order ที่มี Affiliate Link เสร็จสมบูรณ์
- Commission ถูกยกเลิกอัตโนมัติเมื่อ Order ถูกยกเลิก

#### FR-034: ระบบถอนเงินและจ่ายเงิน (Withdrawal & Payout Flow)

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** จัดการการถอนเงินจากหลายแหล่งทั้งแบบ Manual และ Auto พร้อม Batch Payout Process

**ขอบเขตในรุ่นนี้:**

**แหล่งที่มาของ Withdrawal:**

| แหล่งที่มา | ตัวอย่าง | การอนุมัติ |
|-----------|---------|----------|
| **Wallet Withdrawal** | Affiliate ถอน Commission / Seller ถอนจาก Wallet | Manual → Admin Review |
| **Buyer Triggers** | Cancel (จ่ายแล้ว) / Return success / Dispute ชนะ | Auto Approved |
| **Seller Triggers** | Seller cancel (buyer จ่ายแล้ว) / Accept return / Dispute Buyer wins | Auto Approved |

**Admin Payout Process:**
1. Approved → Create Batch → Export to Bank → Bank โอนเงิน → Close Batch
2. Transfer OK → Completed + ออก Payment Voucher (PVW)
3. Transfer Failed → กลับเป็น Approved รอรอบถัดไป

**เกณฑ์ยอมรับ:**

- Wallet Withdrawal ต้องผ่าน Admin Review
- Buyer/Seller Triggers ถูก Auto Approve โดยอัตโนมัติ
- Batch Payout สามารถ Export to Bank ได้
- Transfer Failed จะกลับสถานะเป็น Approved อัตโนมัติ

#### FR-035: เอกสารทางการเงิน 6 ประเภท (Financial Documents)

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** ออกเอกสารทางการเงินอัตโนมัติตาม Event — Marketplace ไม่ออกเอกสารขายแทน Seller

**ขอบเขตในรุ่นนี้:**

**เมื่อ Order = Pending:**

| เอกสาร | Trigger | ออกให้ | หมายเหตุ |
|--------|---------|-------|---------|
| **PV** — Platform Discount Voucher | Buyer ใช้ส่วนลดแพลตฟอร์ม | Seller (เก็บใน Admin) | ยกเลิก Order → VOID |
| **RS** — Receipt Shipping | Payment Completed | Internal/Admin | ยกเลิก/คืนสินค้า → VOID |

**เมื่อ Order = Complete:**

| เอกสาร | Trigger | ออกให้ | หมายเหตุ |
|--------|---------|-------|---------|
| **RV** — GP Invoice/Receipt | Order Complete (หัก GP + ค่าธุรกรรม) | Seller (ปริ้นได้) | ยกเลิกหลัง Complete → VOID |
| **RSA** — Shipping Adjustment | Actual Weight > Estimated | Seller (ปริ้นได้) | ไม่มีส่วนต่าง = ไม่ออก |

**เมื่อ Withdraw / Refund:**

| เอกสาร | Trigger | ออกให้ | หมายเหตุ |
|--------|---------|-------|---------|
| **PVW** — Payment Voucher Withdraw | โอนเงินจริงสำเร็จ | Seller/Affiliate (เก็บใน Admin) | — |
| **RST** — Return Shipping Receipt | Refund Completed / ส่งไม่สำเร็จ-ตีกลับ | Seller (ปริ้นได้) | คืนเงินอย่างเดียว → ค่าส่งขาไป / คืนสินค้า → 2 ขา |

**ข้อจำกัด:** แก้ไขตัวเลขไม่ได้ / Seller เห็นเฉพาะของตน / Trace กลับ Event ได้ / Void = Watermark แต่ยังดูได้

**เกณฑ์ยอมรับ:**

- เอกสารแต่ละประเภทออกอัตโนมัติตาม Trigger Event ที่กำหนด
- เอกสาร VOID แสดง Watermark แต่ยังสามารถดูได้
- Seller สามารถดาวน์โหลดเอกสารของตนเองเป็น PDF ได้
- Admin เห็นเอกสารทุกรายการ

#### FR-036: ระบบจัดการเอกสาร (Document Management System)

**ลำดับความสำคัญ:** Must Have

**วัตถุประสงค์:** ศูนย์กลางเอกสารทางการเงิน — Admin เห็นทุกเอกสาร / Seller เห็นเฉพาะของตน

**ขอบเขตในรุ่นนี้:**

- **Flow:** เข้าเมนู → Filter/Search → View Detail → Download PDF หรือ Void
- **คอลัมน์:** Doc No., Doc Type, Issued To, Order ID, Amount, Status, Date
- **Filter:** Doc Type, Issued To, Date Range, Order ID, Doc No., Amount
- **Trigger Events:** Order Complete → RV/PV, Refund → PVW/RST, Shipping Adj. → RSA
- **Constraints:** แก้ไขตัวเลขไม่ได้, Seller เห็นเฉพาะของตน, Trace กลับ Event ได้

## 9. Monetization & Fee Structure

### 9.1 ค่าธรรมเนียมแพลตฟอร์ม (Platform Fee / GP)

- **อัตรา:** กำหนดโดย Admin ตามหมวดหมู่สินค้า (สามารถตั้งค่าและเปลี่ยนแปลงได้ผ่านระบบหลังบ้าน)
- **ฐานคำนวณ:** ยอดหลังหักส่วนลดร้านค้า
- **การเรียกเก็บ:** เรียกเก็บเฉพาะ Order = Complete เท่านั้น

### 9.2 ค่าธุรกรรมการชำระเงิน (Payment Transaction Fee)

- **อัตรา:** กำหนดโดย Admin (ขึ้นอยู่กับ Payment Gateway)
- **ฐานคำนวณ:** ยอดที่ลูกค้าชำระ
- **กรณี Cancel:** Platform รับภาระแทน ไม่เรียกเก็บจาก Seller

### 9.3 ค่าคอมมิชชัน Affiliate

- **อัตรา:** ผู้ขายกำหนดเองแยกตามรายสินค้า
- **ฐานคำนวณ:** ยอดหลังหักส่วนลดร้านค้า
- **เงื่อนไข:** เรียกเก็บเมื่อ Order Completed + มี Affiliate Attribution

### 9.4 ค่าส่ง (Shipping Cost)

- **Markup:** ส่วนต่างระหว่าง Estimated vs Actual Weight
- **Actual > Estimated:** หักจาก Seller Wallet + ออก RSA Document
- **Actual < Estimated:** ส่วนต่าง = BuyDi Revenue

## 11. หมายเหตุสำคัญ

1. **Escrow Period = 3 วัน** — อัปเดตจาก PRD v1.0 ที่ระบุ 7 วัน ให้ตรงกับ Production ปัจจุบัน
2. **ค่าธรรมเนียม** ทุกรายการสามารถตั้งค่าและเปลี่ยนแปลงได้ผ่านระบบหลังบ้าน Admin ไม่มีตัวเลข % คงที่ในระบบ
3. **BuyDi มีสถานะเป็น E-Marketplace** — ตัวกลางในการถือครองเงิน (Escrow) ไม่ออกเอกสารขายแทน Seller
4. **Wallet มีเฉพาะ Seller และ Affiliate** — ไม่มี Wallet สำหรับ Buyer
5. **Admin ตัดสิน Dispute ถือเป็นที่สิ้นสุด**
6. **Phase 2:** แอพมือถือ (Buyer + Seller) และแชท/ข้อความ ยังไม่อยู่ในขอบเขต Phase 1
7. **Permanently Out of Scope:** ระดับสมาชิก/subscription สำหรับผู้ขาย

## 12. Assumptions

**พฤติกรรมผู้ใช้:**

- ผู้ขาย SME กำลังมองหาทางเลือกที่โปร่งใสเป็นทางเลือกแทนตลาดกลางหลักเช่น Shopee และ Lazada
- ผู้ขายให้ความสำคัญกับเครื่องมือดำเนินงานที่แม่นยำ (SKU, variations, สต็อก) และความโปร่งใสทางการเงิน
- ผู้ขายจะลงทุนเวลาเพื่อปรับแต่งร้านค้าโดยใช้ HTML editor

**การดำเนินงาน:**

- ทีม Admin มีความพร้อมในการรักษา SLA การอนุมัติ 24-48 ชั่วโมงอย่างสม่ำเสมอ
- ทีม Admin สามารถจัดการปริมาณการตรวจสอบและอนุมัติเนื้อแพลตฟอร์มเมื่อขยายตัว

**โครงสร้างพื้นฐานทางเทคนิค:**

- Payment gateways สามารถจัดการปริมาณธุรกรรม, การทักค่าคอมมิชชัน และการจ่ายเงินไปยังบัญชีธนาคารผู้ขายได้อย่างถูกต้อง
- โครงสร้างพื้นฐานสามารถปรับขนาดเพื่อรองรับปริมาณธุรกรรมที่คาดหวังและผู้ใช้พร้อมกัน (10,000+)
- เป้าหมาย uptime 99.9% สามารถทำได้ด้วยโครงสร้างพื้นฐานและการตรวจสอบปัจจุบัน

**การปฏิบัติตาม:**

- กระบวนการตรวจสอบผู้ขาย (ข้อกำหนดหนังสือรับรองบริษัท/บัตรประชาชน) ตรงตามข้อกำหนดทางกฎหมายไทย
- ปฏิบัติตาม PDPA ผ่านการจัดเก็บ PII ที่เข้ารหัส, การเข้ารหัสเอกสาร, และระบบบันทึก audit
